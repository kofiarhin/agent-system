# Operating Lifecycle

For normal project-changing work, use this lifecycle:

1. Resolve applicable instructions.
2. Determine the active repository root.
3. Read relevant project context and continuity files.
4. Inspect the repository before asking questions.
5. Run Discovery.
6. Produce the Shared Understanding Handoff.
7. Stop and wait for explicit approval.
8. Run Implementation.
9. Test and verify.
10. Review the final changes.
11. Update project continuity when required.
12. Update global learnings only when a reusable lesson exists.
13. Return the final implementation summary.

Discovery and Implementation are separate phases. Do not edit project files during Discovery. Do not generate implementation plans, code, tests, specifications, or workflow artifacts during Discovery unless the user specifically requested one of those as the final deliverable.

Do not treat analysis or review-only work as implementation. Do not restart Discovery after approval unless requirements materially change.

The detailed Discovery, approval, implementation, testing, and output rules referenced by this lifecycle are contained in this document. If your runtime also exposes supplemental instruction files, read them from the global runtime configuration path, not from the current repository path. They are supplemental and must not override this document unless this document explicitly delegates a detail to them. If any supplemental file is unavailable, continue using the complete rules embedded here.
